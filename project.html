Building a Surgical Robot Simulation from ROS 2 Theory to Sensors & Control

This post documents the first three milestones: building the ROS 2 foundation, modeling the robot in URDF/Xacro, and making the robot move in Ignition Gazebo with ros2_control, differential drive, odometry, and LiDAR data.

The goal of this portfolio project is to build a mobile manipulator in a simulated operating-theater environment. The robot is designed as a surgical-assistive mobile platform that can eventually navigate around sterile zones and transfer instruments between tables. The first three phases focus on the core infrastructure needed before navigation, manipulation, and task planning can be added.

Phase 0 — Foundation Theory

Understanding the ROS 2 communication model

Before writing simulation code, I focused on understanding the ROS 2 system model. ROS 2 is middleware: it sits between application code and hardware, allowing independent processes called nodes to communicate through standardized mechanisms.

Topics

Used for continuous one-way streams such as /scan, /joint_states, /cmd_vel, and /odom.

Services

Used for short request-response operations, such as saving a map or checking whether a command succeeded.

Actions

Used for long-running tasks with feedback and cancellation, such as navigation goals or arm motion planning.

ROS 2 decision framework

Is the data continuous and one-directional?
    YES → Topic

Does the caller need a response?
    YES → Is the task short and fast?
        YES → Service
        NO  → Does it need feedback or cancellation?
            YES → Action

TF2: the spatial backbone of the robot

TF2 tracks where every robot part is relative to every other part. Every link has a coordinate frame, and every joint defines a transform between parent and child frames. Navigation, visualization, mapping, and manipulation all depend on this tree. 

map
 └── odom
      └── base_footprint
           └── base_link
                ├── left_wheel_link
                ├── right_wheel_link
                ├── caster_link
                ├── lidar_link
                └── torso_link
                     └── shoulder_link
                          └── upper_arm_link
                               └── forearm_link
                                    └── wrist_link
                                         └── gripper_link
Key learning:

robot_state_publisher needs two inputs: the URDF, which defines geometry and joint relationships, and /joint_states, which provides the current joint positions. Together, they generate the dynamic TF2 tree.

Modeling rules learned

URDF inertia matters

Incorrect inertia values can cause unstable simulation behavior. The most important rule is that each inertia term excludes its own axis: ixx uses y² + z², not x².

Wheel orientation matters

Cylinders are vertical by default in URDF. A wheel must be rotated with rpy="1.5708 0 0" so its axle aligns correctly.

base_footprint is the ground frame

base_footprint is projected onto the ground plane and is used by navigation systems for robot footprint and collision reasoning.

Xacro avoids duplication

Xacro properties define values like wheel radius and separation once, then reuse them everywhere using ${name}.

Phase 1 result:

The robot model was verified in RViz2 with 13 TF frames, correct wheel placement, visible LiDAR and arm structure, and a clean base-to-arm hierarchy.

Phase 2 — Sensors & Control

Making the robot move in Ignition Gazebo

Phase 2 transformed the robot from a static RViz model into a controlled simulation. The robot can spawn into Ignition Gazebo, accept velocity commands, move using a differential-drive controller, publish odometry, and stream LiDAR data into ROS 2.

ros2_control architecture

Controller layer
    diff_drive_controller
    joint_state_broadcaster
          │
          ▼
controller_manager
    loads controllers
    manages interface access
    prevents command conflicts
          │
          ▼
hardware interface layer
    Ignition Gazebo: ign_ros2_control plugin
    Real robot: motor driver / hardware driver

Command interface

A write channel. The controller sends a target value to the joint, such as wheel velocity in rad/s.

State interface

A read channel. The controller or broadcaster reads the current joint position, velocity, or effort.

Differential drive math

The diff_drive_controller converts a velocity command into left and right wheel speeds: 

left_wheel_velocity  = (linear.x - angular.z × wheel_separation / 2) / wheel_radius
right_wheel_velocity = (linear.x + angular.z × wheel_separation / 2) / wheel_radius
linear.x angular.z Robot behavior
0.5 0.0 Straight forward
0.0 0.8 Rotate in place to the left
0.5 0.8 Curve left while moving forward
-0.5 0.0 Straight backward

LiDAR pipeline

Ignition Gazebo publishes sensor data using Ignition message types. To use that data in ROS 2, it must be bridged through ros_gz_bridge.

Important command topic detail

Because use_stamped_vel: false was used, the controller listens on /diff_drive_controller/cmd_vel_unstamped, not directly on /cmd_vel. Teleoperation therefore needs topic remapping: 

ros2 run teleop_twist_keyboard teleop_twist_keyboard --ros-args \
  -r /cmd_vel:=/diff_drive_controller/cmd_vel_unstamped
Simulation timing rule:

Every node in the simulation must use use_sim_time: True. Mixing Gazebo simulation time with wall-clock time causes TF2 lookup failures and stale transforms.

Debugging log

Errors encountered and how they were fixed

Phase 2 was not only about getting the robot working — it was also about understanding why things fail. These were the key issues found during integration.

Problem Root Cause Fix
Robot did not move Commands were published to /cmd_vel, but the controller listened on /diff_drive_controller/cmd_vel_unstamped. Publish to the correct topic or remap teleop output.
Gazebo physics did not behave correctly IGN_GAZEBO_SYSTEM_PLUGIN_PATH was manually overwritten, breaking Gazebo plugin discovery. Remove manual override and use the standard ros_gz_sim launcher.
Controller configuration did not load correctly controllers.yaml was referenced through a hard-coded source path. Use $(find surgical_robot_description)/config/controllers.yaml so the installed path resolves at runtime.
LiDAR did not publish data ogre2 render engine required GPU acceleration, so the sensor system loaded but generated no data. Use <render_engine>ogre</render_engine>.
Robot slipped or moved poorly Default wheel friction/contact settings were too weak. Add mu1, mu2, kp, and kd to wheel Gazebo tags.
Wheel direction was wrong Wheel joint axis sign did not match the controller's forward-motion convention. Change wheel axis to <axis xyz="0 0 -1"/>.
Odometry bridge confusion /diff_drive_controller/odom was bridged from Ignition even though it is already produced directly by ROS 2. Bridge only Ignition-produced topics such as /clock and /scan.
