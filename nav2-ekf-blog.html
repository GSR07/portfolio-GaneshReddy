Nav2 EKF Sensor Fusion on Custom AMRs

Integrating wheel odometry, IMU data, and 2D LiDAR into the robot_localization package for a 300kg payload autonomous mobile robot.

1. The Navigation Challenge

Building a 300kg payload AMR is demanding on the hardware level, but the software challenge is equally difficult. A heavy robot experiences slippage, wheel scrub, and IMU drift that can completely disorient its navigation stack. Standard wheel odometry alone is not reliable. If you rely solely on it, the robot will quickly believe it's in a different room.

To fix this, we need Sensor Fusion using an Extended Kalman Filter (EKF) to merge multiple imprecise measurements into a single, high-fidelity state estimate.

flowchart TD Odom[Wheel Odometry] -->|Velocity| EKF[robot_localization EKF Filter] IMU[IMU Sensor] -->|Angular Velocity, Acceleration| EKF Lidar[2D LiDAR] -->|Laser Scans| AMCL[AMCL] AMCL -->|Pose Estimate| EKF EKF -->|Fused Odom to Base Link| Nav2[Nav2 Stack]

2. Understanding the Extended Kalman Filter (EKF)

The standard Kalman Filter works perfectly for linear systems, but our AMR model is non-linear (due to trigonometric functions representing headings). The EKF solves this by taking the Jacobian of the state transition and observation models.

\[ \hat{x}_{k|k-1} = f(\hat{x}_{k-1|k-1}, u_k) \] \[ P_{k|k-1} = F_k P_{k-1|k-1} F_k^T + Q_k \]

Predict Phase: The robot guesses its current position based on its previous position and how much it thinks its wheels have turned (\(u_k\)).

\[ K_k = P_{k|k-1} H_k^T (H_k P_{k|k-1} H_k^T + R_k)^{-1} \] \[ \hat{x}_{k|k} = \hat{x}_{k|k-1} + K_k (z_k - h(\hat{x}_{k|k-1})) \]

Update Phase: The robot receives new sensor data, like an IMU reading (\(z_k\)), and applies the Kalman Gain (\(K_k\)) to correct the state estimate based on how much it trusts the sensor (\(R_k\)) vs its own prediction.

3. Configuring robot_localization

In ROS2, we use the robot_localization package to handle the EKF math. The trick is telling it exactly which parts of which sensor messages to trust.

ekf_filter_node:
  ros__parameters:
    frequency: 30.0
    two_d_mode: true
    publish_tf: true

    # Wheel Odometry
    odom0: /odom
    odom0_config: [false, false, false,
                   false, false, false,
                   true,  true,  false,
                   false, false, true,
                   false, false, false]
                   
    # IMU Data
    imu0: /imu/data
    imu0_config: [false, false, false,
                  false, false, true,
                  false, false, false,
                  false, false, true,
                  true,  true,  false]

Notice the boolean arrays. We instruct the EKF to only look at X/Y velocity and Yaw velocity from the wheel odometry, but rely heavily on the IMU for Yaw orientation and angular velocity. This ensures we get the best traits of both sensors while mitigating their individual weaknesses.

4. Nav2 Integration & Results

Once the EKF publishes the high-fidelity odom to base_link transform, we plug it into the Nav2 stack alongside AMCL (Adaptive Monte Carlo Localization) which provides map-level corrections based on LiDAR.

The final result is an incredibly stable AMR that can recover from slippage on uneven terrain and accurately trace its path on the map with near-zero long-term drift.
